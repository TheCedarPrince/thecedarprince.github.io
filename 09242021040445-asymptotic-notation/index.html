Big-O Notation

Overview

  • Big-O (denoted as O\mathcal{O}O) refers to the most accurate worst case analysis with regards to either

- Execution time
- Space used (in memory or disk)

  • It always assumes the maximum number of iterations

Conventions

  • Constants get dropped from O\mathcal{O}O notation as it is small compared to infinity

Example:

O(5n)→O(n) \mathcal{O}(5n) \rightarrow \mathcal{O}(n) O(5n)→O(n)

  • Lower order terms get dropped in O\mathcal{O}O notation for the reason as before

Example:

O(n2+1000n−3)→O(n2) \mathcal{O}(n^{2} + 1000n - 3) \rightarrow \mathcal{O}(n^{2}) O(n2+1000n−3)→O(n2)

  • Dropping constants theoretically is possible because constants do not grow towards infinity

  • In practice however, these constants can affect practical outcomes of algorithms

Constant Complexity: O(1)\mathcal{O}(1)O(1)

  • Performance does not scale with input size

  • Example is having a list and returning the first item in the list:

mylist = [1:5...]
first(mylist)

Linear Complexity: O(n)\mathcal{O}(n)O(n)

  • Performance does scale with size

  • Example is summing all elements in an array:

mylist = [1:5...]
summed_values = sum(mylist)

Logarithmic Complexity: O(log(n))\mathcal{O}(log(n))O(log(n))

  • Performance scales logarithmically with input size

  • Base doesn't matter due to change of base:

logm(n)=log2(n)log2(m)=Clog2(n)→O(logm(n))→O(log(n)) log_{m}(n) = \frac{log_{2}(n)}{log_{2}(m)} = Clog_{2}(n) \rightarrow \mathcal{O}(log_{m}(n)) \rightarrow \mathcal{O}(log(n)) logm​(n)=log2​(m)log2​(n)​=Clog2​(n)→O(logm​(n))→O(log(n))

Exercises

The following exercises can be found here

  1. What is the time and space complexity of:

a = 0
i = 0

while i < N
	a = a + rand(1)
	i += 1
end


b = 0
j = 0

while j < M
	b = b + rand(1)
	j += 1
end

Answer: O(N+M),O(1)\mathcal{O}(N + M), \mathcal{O}(1)O(N+M),O(1)

Explanation: Since we measure complexity by worse case scenario of primitive operations ran, there could be NNN and MMM operations executed in this code. As no additional space is being utilized, space complexity is constant as no new variables are being defined.

  1. What is the time complexity of:

a = 0
i = 0
j = N

while i < N
	while j > i
		a = a + i + j
		j -= 1
	end
	i += 1
end

Answer: O(N⋅N)\mathcal{O}(N \cdot N)O(N⋅N)

Explanation: Both loops are dependent on NNN so both loops, iterate NNN times, therefore resulting in a time complexity of O(N⋅N)\mathcal{O}(N \cdot N)O(N⋅N)

  1. What is the time complexity of the following code:

i = N / 2
k = 0

while i <= N
	j = 2
	while j <= N
		k = k + N / 2
		j *= 2
	end
	i += 1
end

Answer: O(n⋅log⁡(n))\mathcal{O}(n \cdot \log(n))O(n⋅log(n))

Explanation: As nnn continues to increase, the variable, kkk continues to loosely grow more than exponential. Furthermore, there are, n2\frac{n}{2}2n​ primitive steps in the outer loop such that the total time complexity would be O(n2⋅log⁡n)\mathcal{O}(\frac{n}{2} \cdot \log{n})O(2n​⋅logn) which is then simplified to O(n⋅log⁡(n))\mathcal{O}(n \cdot \log(n))O(n⋅log(n)) I got this wrong initially because I did not account for the outer loop contributing a time complexity of n2\frac{n}{2}2n​.

  1. What does it mean when we say that algorithm X is asymptotically more efficient than Y?

Answer: Algorithm X will always be better for large inputs

Explanation: When we consider an asymptote in terms of an algorithm, we also consider that algorithms "growth" over time. Meaning, that if you have some algorithm that is efficient at an asymptote, by nature of asymptotic analysis, that means it is "good" in the worst case scenario of that algorithm.

Addendum: I got this wrong when thinking about asymptotic notation as I failed to consider growth. I thought X would be better for all inputs to that algorithm but that would not be so in the case of possibly a smaller input to X.

  1. What is the time complexity of the following code:

a = 0
i = N

while i > 0
	a += i
	i /= 2
end

Answer: O(log⁡n)\mathcal{O}(\log{n})O(logn)

Explanation: There is a direct proportional relationship between nnn and the final output.

  1. What best describes the useful criterion for comparing the efficiency of algorithms?

Answer: Time and Memory

Explanation: Time dictates how long a program will evaluate for and memory dictates how much a program can evaluate

  1. How is time complexity measured?

Answer: By counting the number of primitive operations in an algorithm on a given input size.

Explanation: Each primitive operation is generally assumed to evaluate at the cost of "one" for each operation.
